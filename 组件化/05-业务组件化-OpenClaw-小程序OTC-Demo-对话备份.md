## 1. 背景与转型方向（摘要）

- **角色**：医药电商 B2C 业务负责人；App：**1药网**；模式：垂直 B2C。  
- **痛点**：垂直电商流量承压，需转型。  
- **方向一：业务组件化**  
  - 将「可卖药、可履约」拆为**边界清晰、可对外签约（contract）的能力模块**。  
  - 渠道可包括自有 App、小程序、合作方、以及 **AI Agent / Skill**（社区常将 **OpenClaw** 昵称为「小龙虾」）。  
  - **OpenClaw 参考**：[openclaw/openclaw](https://github.com/openclaw/openclaw)；插件/Skill 脚手架可参考社区资料如 [create-plugin skill](https://playbooks.com/skills/openclaw/skills/create-plugin)。

---

## 2. 「组件化」三层结构

| 层级 | 含义 |
|------|------|
| **业务能力组件** | 商品与可售性、订单、支付、履约、会员与营销等（Rx 后置） |
| **集成组件（API / 事件 / SDK）** | 鉴权、幂等、回调、限流、沙箱 |
| **渠道封装（Skill / Plugin / MCP 等）** | 将 API 包成某生态可调用的「技能」 |

**原则**：Agent 侧多做「自然语言 → 结构化意图 → 调能力」；**医药交易与合规校验收口在可控系统**，避免「模型直接下单」。

## 3. 已确认的约束与决策

| 项 | 结论 |
|----|------|
| **OpenClaw** | 即社区所称「小龙虾」框架；对接以 **HTTPS Tool / 插件** 调用己方 BFF 为宜，不用桌面自动化替代交易系统。 |
| **首发渠道** | **微信小程序**（已有现网小程序，**支付与下单流程已跑通**）。 |
| **品类** | **先做 OTC**；处方后续再做；**不自建互联网医院**（Rx 阶段需另议合作方/处方来源）。 |
| **内部系统** | 商品与订单 **多套**；**当前无可直接调用的 HTTP 接口**。 |
| **首月目标** | **约 1 个月内**第一个 **可真实微信支付** 的 Demo。 |
| **组织** | 有 **专职运营**，**无专职开放平台团队** —— 采用小队混编（业务 + 运营 + 后端 + 小程序 + 内部系统接口人）。 |
| **用户路径** | **商详 → 加购 → 结算**（不在首月强求 Agent 远程调「加购接口」；**Deep link 到商详**最稳）。 |
| **部署与权限** | **顾客谁都可以买**（在小程序内完成交易）≠ **BFF 对公网无鉴权开放**；BFF 需 **Token + 限流 + 审计**；OpenClaw 仅持密钥侧调用。 |
| **对内 / 对外** | **同一套叙事**；对外可用 **录屏 + 固定演示 SKU + 只读/演示 Key**。 |

## 4. 架构共识（MVP）

```
用户（微信） → 现网小程序：商详 → 加购 → 结算 → 真实微信支付
                    ↑
              Deep link（path + query，如 skuId、from=openclaw）

OpenClaw（可选同月） → BFF（新建，对外唯一入口）→ OTC Demo 商品索引 / 白名单
                    （不替代订单中心；首月不强求实时打通所有内部库）
```

**无接口时的首月策略**：  
- **白名单 10～30 OTC SKU**（运营维护表）；或  
- **只读同步层**（导出/只读从库/定时任务）→ 小型「索引服务」供 `search` / `skuDetail`。

## 5. 四周排期（摘要）

| 周 | 目标 |
|----|------|
| W1 | Demo 边界、小程序落地参数协议、BFF 仓库与鉴权设计 |
| W2 | BFF v0：`search`、`skuDetail`、返回小程序打开所需 `path`/`query` |
| W3 | OpenClaw 插件联调：对话 → 打开小程序 → 真实支付 |
| W4 | 演示脚本、对内发包、对外材料（录屏 + 文档） |

**待研发确认**：现网商详是否已支持 **URL 参数直达 SKU**；若否，增加 **落地中转页** 再 `redirectTo` 商详。

## 6. 与「上次 GitHub 备份成功」的类比（团队内沟通用）

- 早期失败多因 **目录不对 / 无 remote / 认证方式错误**。  
- **成功路径**：在 **已有、已配置 remote 的仓库** 内 **`add` → `commit` → `push`**。  
- 本业务同理：**复用现网小程序交易链** + **新建薄 BFF**，避免首月重做支付与订单中台。
